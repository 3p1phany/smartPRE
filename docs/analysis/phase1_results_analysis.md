## 1. 实验回顾

### 1.1 机制概述

RE-driven Timeout Escalation 将 RE Store 的被动拦截转为主动调整：当某 RE 条目连续命中 N 次后，直接提升该 bank 的 timeout 到更高档位，并释放 RE 条目以腾出容量。设计目标是形成负反馈闭环：

```
RE 高命中 → 提升 timeout → 减少错误 precharge → RE 命中减少 → 释放 RE 容量
```

同时，timeout 候选集从 7 个 `{50, 100, 150, 200, 300, 400, 800}` 扩展为 9 个 `{50, 100, 150, 200, 300, 400, 800, 1600, 3200}`，使 shadow simulation 可选择更高档位。

### 1.2 参数配置

固定 DECAY_PERIODS=5, CONFLICT_THR=3, ALLOW_OPEN=false，扫描 N（连续命中阈值）和 STEP（提升步长）：

| 标签 | N | STEP | 设计意图 |
|------|---|------|---------|
| GS_esc_N3_S1 | 3 | 1 | 激进触发，保守提升 |
| GS_esc_N3_S2 | 3 | 2 | 激进触发，中等提升 |
| GS_esc_N5_S1 | 5 | 1 | 中等触发，保守提升 |
| GS_esc_N5_S2 | 5 | 2 | 中等触发，中等提升 |
| GS_esc_N5_S3 | 5 | 3 | 中等触发，激进提升 |
| GS_esc_N8_S1 | 8 | 1 | 保守触发，保守提升 |
| GS_esc_N8_S2 | 8 | 2 | 保守触发，中等提升 |
| GS_esc_N12_S2 | 12 | 2 | 极保守触发 |

### 2.3 相对 Open-Page 的 Speedup

GS 策略的核心价值是相对纯 open-page 的性能优势。以下对比各配置 vs open_page baseline 的 GEOMEAN speedup：

| 配置 | vs open_page | vs GS baseline | 说明 |
|------|-------------|----------------|------|
| GS_1c (baseline) | **+1.53%** | — | GS 原始实现 |
| GS_NOHOTROW_1c | -1.62% | -3.15% | 无 RE Store 消融 |
| GS_esc_N12_S2 | +1.43% | -0.10% | 最佳 escalation |
| GS_esc_N8_S2 | +1.42% | -0.12% | |
| GS_esc_N5_S2 | +1.40% | -0.13% | |
| GS_esc_N5_S3 | +1.39% | -0.14% | |
| GS_esc_N3_S2 | +1.38% | -0.15% | |
| GS_esc_N8_S1 | +1.35% | -0.18% | |
| GS_esc_N5_S1 | +1.32% | -0.21% | |
| GS_esc_N3_S1 | +1.30% | -0.23% | 最差 escalation |

所有 escalation 配置仍优于 open_page（+1.30~1.43%），但均未超过 GS baseline 的 +1.53%。Escalation 削弱了 GS 相对 open_page 的优势。

### 2.5 Open-Page 对比关键观察

**1. Escalation 对 GS 已有的强优势 benchmark 影响不一：**

**2. GS baseline 对 open_page 弱于或持平的 benchmark 中，部分被 escalation 改善：**

## 3. 参数敏感性分析

### 3.1 N（连续命中阈值）的影响

固定 STEP=2 对比不同 N：

| N | GEOMEAN | Wins | Losses |
|---|---------|------|--------|
| 3 | -0.15% | 15 | 30 |
| 5 | -0.13% | 16 | 27 |
| 8 | -0.12% | 15 | 27 |
| 12 | -0.10% | 15 | 28 |

**规律：N 越高 regression 越小。** 更高的 N 意味着更少的 escalation 触发。当 N→∞ 时退化为纯 GS baseline（零 escalation），这表明 escalation 机制本身是 net harmful 的——触发越少越好。

### 3.2 STEP（提升步长）的影响

固定 N=5 对比不同 STEP：

| STEP | GEOMEAN | Wins | Losses |
|------|---------|------|--------|
| 1 | -0.21% | 11 | 39 |
| 2 | -0.13% | 16 | 27 |
| 3 | -0.14% | 16 | 26 |

**规律：STEP=1 显著劣于 STEP=2/3。** 原因是 STEP=1 释放了 RE 条目但 timeout 提升太小（如从 800c 到 1600c），不足以弥补失去的 RE 被动拦截保护力度。STEP=2 和 STEP=3 表现接近，但 STEP=3 在部分 benchmark（如 crono/PageRank）上因过高 timeout 引入更多 conflict。

### 3.3 参数交互效应

各配置在 winner 和 loser 集上的表现模式：

- **Winner 集**（Triangle-Counting、omnetpp 等）：STEP=2/3 优于 STEP=1，N 影响较小。改善可能来自 shadow simulation 利用扩展后的候选集（1600c、3200c），而非 escalation 机制本身。

- **Loser 集**（PageRank/soc-pokec、hashjoin 等）：N 越小损失越大（更多 escalation 触发），STEP=1 也更差。Regression 与 escalation 触发频率正相关。

## 7. 核心发现：RE 被动拦截的隐含价值

Phase 1 最重要的发现是 **RE Store 被动拦截机制的实际保护力度远超预期**。

### 7.1 设计假设 vs 现实

| 设计假设 | 现实 |
|---------|------|
| RE 反复拦截是"无谓循环"，浪费容量 | RE 拦截实质上提供了**无限 hold time**——只要条目在 RE Store 中，行就不会因 timeout 被关闭 |
| Escalation 释放条目后用显式 timeout 替代 | 最高 timeout 3200c 远不及 RE 拦截的等效 hold time（RE 命中倍率 18x × 800c ≈ 14400c 等效保护） |
| 释放 RE 容量可以保护更多行 | 大部分 benchmark 不存在 RE 容量瓶颈（eviction 接近零） |

### 7.2 负反馈闭环未按预期工作

设计预期的闭环：
```
RE 高命中 → escalation → timeout 提高 → RE 命中减少 → RE 插入减少
```

实际观测：
```
RE 高命中 → escalation → RE 条目被释放 → 保护丧失 → 更多 timeout precharge 错误 → RE 插入量不变
```

RE 插入量（即 timeout 错误数）在所有配置下基本不变，表明 escalation 后的 timeout 仍然不够大，无法避免后续的 timeout precharge 错误。负反馈闭环的"减少 RE 插入"环节未实现。

## 8. 未解答的关键问题

所有 8 个配置共享了一个变更：timeout 候选集从 7 扩展到 9（新增 1600c、3200c）。获益 benchmark（Triangle-Counting、omnetpp 等 12 个）的改善 **可能完全来自 shadow simulation 利用扩展候选集**，而非 escalation 机制。

佐证：
- STEP=1 configs（S1）几乎全面劣于 STEP=2/3（S2/S3）。S1 的 escalation 更"保守"但表现更差，暗示 escalation 本身不是改善来源。
- 获益 benchmark 在 S2 和 S3 间表现一致，但 S2 触发的 escalation 比 S3 少——说明改善与 escalation 触发频率无关。
- lbm 的 DRAM 统计显示 1600c+3200c 占 57%，这些选择可能来自 shadow simulation 自主仲裁而非 escalation 强制提升。

**消融实验 `GS_ext_timeout_only`（仅扩展候选集、不做 escalation）可以确认这一假设。** 如果该配置与 escalation 最佳配置表现一致甚至更好，则证明 escalation 的增量贡献为零。

| **N5_S3** | Winner 集天花板最高（omnetpp +0.92%）| Loser 集损失也最大（PageRank -1.50%）|
| **N8_S2** | GEOMEAN regression 最小（-0.12%），Winner/Loser 较均衡 | Winner 集天花板略低 |

### 9.2 建议优先跑的消融实验

在进入 Phase 2（open-page escalation）之前，建议先跑一个消融实验确认 escalation 的增量价值：

- **`GS_ext_timeout_only`**：仅将候选集从 {50..800} 扩展到 {50..3200}，不做任何 escalation 逻辑。这可以分离 "扩展候选集" 和 "RE-driven escalation" 两个变量的各自贡献。

如果 `GS_ext_timeout_only` 表现与 escalation 最佳配置持平或更好，则说明 escalation 本身无增量价值，后续方向应转向：
- 增大 RE Store 容量（128/256）
- 优化 RE 条目替换策略
- 扩展 timeout 候选集（可能进一步加入 6400c 或更多档位）

如果 `GS_ext_timeout_only` 明显更差，则 escalation 有真实贡献，可继续 Phase 2 探索 open-page escalation。
