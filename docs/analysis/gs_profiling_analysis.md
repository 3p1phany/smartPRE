# GS 性能计数器分析报告（GS_prof_1c）

**日期：** 2026-02-26
**数据来源：** `champsim-la/results/GS_prof_1c/`
**规模：** 62 benchmarks, 799 slices, 8 suites (crono, graph500, hashjoin, hpcc, ligra, npb, spec06, spec17)

---

## 1. 数据完整性验证

所有交叉验证不变量均通过：

| 不变量 | 结果 |
|--------|------|
| `gs_re_insertions == gs_timeout_wrong` | 9,054,336 == 9,054,336 PASS |
| `correct + wrong <= precharges` | 95,497,383 <= 95,510,541 PASS（仅 13,158 未验证，0.01%） |
| `re_hit_useful + re_hit_useless <= re_hits` | 63,526,072 <= 1,143,291,884 PASS |
| `re_insertions >= re_evictions` | 9,054,336 >= 9,003,733 PASS |
| `num_pre_for_demand == ondemand_pres + timeout_pres` | 126,741,295 == 126,741,295 PASS（差值为 0） |

**结论：计数器实现正确可靠。**

## 2. 全局总览

### A. Timeout Precharge 准确性

| 指标 | 值 |
|------|-----|
| 总 timeout precharge | 95,510,541 |
| 正确（下次 ACT 不同行） | 86,443,047 (90.51%) |
| 错误（下次 ACT 同一行） | 9,054,336 (9.48%) |
| 未验证（仿真边界/refresh） | 13,158 (0.01%) |

| Timeout PRE 占总 demand PRE 比例 | 75.36% |
| On-demand PRE 占总 demand PRE 比例 | 24.64% |

GS 极大减少了 on-demand precharge，75% 的行关闭由超时投机驱动。

### B. Row Exclusion (RE) Store 有效性

| 指标 | 值 |
|------|-----|
| RE 插入 | 9,054,336 |
| RE 驱逐 | 9,003,733 |

| RE 命中倍率（total hits / unique events） | **18.0x** |
| RE 命中期间服务的 CAS | 39,260,702 |
| 平均每次 useful RE 命中服务的 CAS | 0.87（受限于计数器实现，仅计首次 CAS） |

- RE 命中倍率 18x 表明每个 RE 条目平均被触发 18 次阻止 timeout precharge，RE 保护的行确实是"长寿行"。
- RE Store 几乎饱和（插入 9.05M vs 驱逐 9.00M），64 项容量在许多 workload 下不够用。

## 4. lbm 深度分析

lbm 是 timeout 准确率最低的 benchmark（spec06: 39.1%, spec17: 43.8%），其行为揭示了 GS 机制的核心局限。

### Timeout 分布呈 U 型

```
spec06/lbm:                          spec17/lbm:
  50c: 27.3% ██████████████             50c: 29.4% ███████████████
 100c: 19.7% ██████████               100c: 17.3% █████████
 150c:  6.5% ███                      150c: 12.6% ██████
 200c:  2.6% █                        200c:  2.8% █
 300c:  4.8% ██                       300c:  4.9% ██
 400c:  4.4% ██                       400c:  4.9% ██
 800c: 34.7% █████████████████        800c: 28.2% ██████████████
```

~47% 选短 timeout（50-100c），~35% 选最长 800c，中间档很少。不同 bank 行为差异极大。

### 逐 Slice 行为分化

| 类型 | 典型 Slice | TO_Acc | 800c 占比 | 说明 |
|------|-----------|--------|----------|------|
| 初始阶段 | slice 0 | ~94% | 2-3% | 热身阶段，行局部性低 |
| 特殊阶段 | slice 13/16 | 70-99% | 0-7% | 阶段性行为变化 |
| **稳态（主体）** | slice 1-12, 14+ | **34-36%** | **37-41%** | lbm 核心计算，准确率极低 |

稳态阶段一致表现：TO_Acc ~35%，800c 占比 ~40%，RE_Acc 93%。**Shadow simulation 已将许多 bank 推到 800c 上限但仍不够。**

## 5. lbm-like 行为的全面筛查

### 筛选标准

以 lbm 的核心特征定义 "lbm-like" 行为：
- **Timeout 准确率 < 70%**（timeout 投机频繁出错）
- **800c 占比 > 15%**（shadow simulation 已将大量 bank 推向上限）
- **RE 准确率 > 60%**（RE 机制在有效弥补 timeout 错误）

### 800c 占比与其他指标的相关性

| 800c 占比 | benchmark 数 | 平均 TO_Acc | 平均 RE_Acc | 平均 RBH |
|----------|-------------|------------|------------|---------|
| 0-5% | 17 | 91.5% | 52.1% | 35.1% |

| 10-20% | 19 | 70.5% | 76.6% | 73.9% |

| **30-50%** | **7** | **60.8%** | **86.1%** | **84.1%** |

**规律链条：RBH 高 → 行重访频繁 → shadow simulation 被迫选大 timeout → 800c 仍不够 → TO_Acc 低 → RE 拼命补救 → RE_Acc 高。**

### 三类典型子模式

1. **极端 U 型（lbm、sphinx3、bwaves）**：800c 占比 30-52%，同时短 timeout 也有相当占比。不同 bank 行为差异极大——部分 bank 行局部性低（选短），部分 bank 行长期驻留（已选最大但仍不够）。

2. **roadNet 图类（PageRank/roadNet-CA、Triangle/roadNet-CA 等）**：roadNet-CA 是稀疏公路网图，遍历模式导致某些行被反复访问。分布更均匀，中间档也有可观占比。

3. **科学计算类（wrf、leslie3d、milc）**：stencil/网格计算导致规律性行重访，timeout 准确率 60-80%，RE 有效率 80-92%。

## 6. 总结与建议

### GS 机制整体有效

- 90.5% 的 timeout 准确率，75% 的行关闭由超时投机驱动
- 71.3% 的 RE 准确率，RE 命中倍率 18x 表明热行保护极为活跃
- 对随机访问 workload（hpcc、hashjoin），GS 接近最优 close-page 策略
- Timeout 分布的双峰特征证明 GS 能自适应地在 open-page 和 close-page 间切换

### 核心发现：800c 上限对高行局部性 workload 不够

- 62 个 benchmark 中，**8 个** benchmark 的 800c 是被选择最多的 timeout 值（>30%），**15 个**超过 20%
- **13 个** benchmark 同时表现出 "高 800c 占比 + 低准确率 + 高 RE 依赖"，并非 lbm 独有
- 这些 benchmark 的 RE 机制在高负荷运转弥补 timeout 不足，RE Store 接近饱和

### 改进方向

1. **扩展 timeout 候选集**：在 `{50, 100, 150, 200, 300, 400, 800}` 基础上增加更大值（如 1600c、3200c），使 shadow simulation 能选择更合适的长 timeout。受益 benchmark 不仅限于 lbm，而是涵盖 13-19 个高行局部性 workload。

2. **增大 RE Store 容量**：当前 64 项几乎 100% 饱和（9.05M 插入 vs 9.00M 驱逐）。适度增加容量（如 128 项）可减少有效条目被过早淘汰，尤其对 spec06/mcf（RE 命中 113M 次）等高 RE 活跃度 workload 有益。

3. **RE 连续命中自适应**：当同一 bank 上 RE 连续多次拦截 timeout 时，可考虑自动提升该 bank 的 timeout 值，减少无谓的 timeout 触发与 RE 拦截循环。
